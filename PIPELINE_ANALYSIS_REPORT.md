# Pipeline Analysis Report: G2F Plot-Aware AlphaEarth Embeddings

## Executive Summary

This pipeline extracts **AlphaEarth satellite embeddings** for agricultural research plots from the Global Gene to Phenotype (G2F) dataset. It integrates genomic phenotype data with satellite-derived spatial embeddings to enable environment-phenotype correlation analysis.

---

## 1. Pipeline Architecture Overview

```
Raw G2F Data (CSV)
    ↓
[Preprocessing Stage]
    ├─ Load and validate phenotype CSV
    ├─ Standardize column names and units
    ├─ Convert plot areas (km² → m²)
    ├─ Create unique plot IDs
    ├─ Filter to AlphaEarth-supported years (2017-2024)
    ├─ Validate geographic coordinates
    ↓
Processed Plot Registry (CSV)
    ↓
[AlphaEarth Embedding Extraction]
    ├─ Initialize Google Earth Engine connection
    ├─ Group plots by year
    ├─ Convert plot center + area → square geometry (10m resolution)
    ├─ Query GOOGLE/SATELLITE_EMBEDDING/V1/ANNUAL collection
    ├─ Extract 64-band AlphaEarth embeddings (A00-A63)
    ├─ Compute mean pooling within each plot geometry
    ↓
Output Embeddings (CSV with metadata + 64 embedding dimensions)
```

---

## 2. Pipeline Stages in Detail

### Stage 1: Data Preprocessing (`prepare_plot_input.py`)

**Purpose**: Standardize raw G2F phenotype data into plot-level records ready for satellite data extraction.

**Process**:

1. **Load Configuration**: Read YAML config mapping raw CSV columns to standardized names
   - Maps G2F column names (Field Location, Plot, Year, etc.) to pipeline names

2. **Input Validation**: Verify all required columns exist
   - Required: site_id, plot_number, year, latitude, longitude, plot_area_km2

3. **Column Standardization**: Convert raw columns to pipeline format
   - Standardize data types: site_id (str), year (int), coordinates (float), areas (float)

4. **Unit Conversion**: Convert plot areas from km² to m²
   - Formula: `plot_area_m2 = plot_area_km2 × 1,000,000`
   - Critical for AlphaEarth pixel-level calculations

5. **Unique ID Creation**: Compose plot identifiers from:
   - `plot_id = site_id + "_" + year + "_" + plot_number`
   - Ensures uniqueness across sites and years

6. **Temporal Filtering**: Keep only AlphaEarth-supported years (default: 2017-2024)

7. **Data Validation**: Remove invalid records:
   - NULL values in critical fields
   - Latitude outside [-90, 90]
   - Longitude outside [-180, 180]
   - Plot areas ≤ 0

8. **Expected Pixel Count Calculation**: Estimate sampling coverage
   - AlphaEarth pixel size ≈ 10m × 10m = 100m²
   - `expected_pixel_count = plot_area_m2 / 100.0`

**Output**: Clean CSV with standardized plot records

---

### Stage 2: AlphaEarth Embedding Extraction (`run_plot_area_embeddings.py`)

**Purpose**: Extract satellite-derived embeddings for each plot-year combination.

#### 2.1 Earth Engine Initialization
- Authenticates to Google Earth Engine using `EE_PROJECT` environment variable
- Required: Valid Google Cloud project with Earth Engine API enabled

#### 2.2 Year-Grouped Processing
- Plots grouped by year to align with annual AlphaEarth imagery (GOOGLE/SATELLITE_EMBEDDING/V1/ANNUAL)
- Each year processed independently

#### 2.3 Geometry Construction: `square_from_lat_lon_area()`

**Input**: Plot center coordinate (lat, lon) + area in m²

**Process**:
```
1. Compute square side length: side_m = sqrt(area_m2)
2. Compute half-side: half_side_m = side_m / 2
3. Create point at plot center: ee.Geometry.Point([lon, lat])
4. Buffer point by half_side: buffer(half_side_m)
5. Convert to square bbox: bounds()
```

**Rationale**:
- Plot boundary unknown; only center and area available
- Square approximation reasonable for agricultural plots
- Geometry is coordinate-aligned (not rotated)

#### 2.4 Feature Collection Building

**Purpose**: Convert tabular plot data into Earth Engine spatial features

**Structure**:
```
Feature Properties:
  - plot_id: Unique identifier
  - year: Year of observation
  - latitude, longitude: Plot center
  - plot_area_m2: Total area
  - expected_pixel_count: Sampling coverage estimate
  - geometry_method: "square_from_plot_area"
  - geometry: Spatial bounds (coordinate system: EPSG:4326)
```

#### 2.5 AlphaEarth Band Selection

**Available Bands**: 64 embedding dimensions named A00 through A63
- Generated by `alphaearth_band_names()`: returns `["A00", "A01", ..., "A63"]`
- Each band represents a learned feature from satellite imagery

#### 2.6 Embedding Extraction: `extract_embeddings_for_year()` — DETAILED STEP-BY-STEP

**High-Level Process**:
1. Filter AlphaEarth annual collection to target year (Jan 1 - Dec 31)
2. Create mosaic of all images in year (handles cloud/gaps)
3. Select 64 embedding bands (A00-A63)
4. Apply mean pooling over each plot's geometry
5. Return aggregated values

---

### 2.6.1 Step 1: Time-Window Filtering

**Purpose**: Extract satellite data for a specific calendar year only.

**Code Logic**:
```python
start_date = ee.Date.fromYMD(year, 1, 1)      # Jan 1 of target year
end_date = start_date.advance(1, "year")      # Jan 1 of next year
```

**Example (Year 2020)**:
- `start_date` = 2020-01-01
- `end_date` = 2021-01-01
- Earth Engine then filters collection to images acquired between these dates

**Why This Matters**:
- AlphaEarth collection has one annual composite per year
- Filtering by date ensures you get the correct year's data
- Different years may have different cloud patterns, sensor availability

---

### 2.6.2 Step 2: Image Collection & Mosaicking

**Purpose**: Handle multiple satellite images in a year and create a single unified image.

**Code**:
```python
image = ee.ImageCollection(collection_name).filterDate(start_date, end_date).mosaic()
```

**What Happens**:
```
ImageCollection for Year 2020:
  ├─ Image 2020-01-15 (from satellite pass #1)
  ├─ Image 2020-03-22 (from satellite pass #2)
  ├─ Image 2020-06-10 (from satellite pass #3)
  ├─ Image 2020-09-05 (from satellite pass #4)
  └─ Image 2020-11-30 (from satellite pass #5)

mosaic() → Combines all into ONE image by:
  - Stacking images
  - Taking first valid (non-cloud) pixel from top
  - Result: Single annual composite image per location
```

**Visual Example**:
```
5 separate satellite observations (cloud masks)

Image 1: [C][C][V][C][C]      C = Cloud, V = Valid
Image 2: [V][C][C][V][C]
Image 3: [C][V][C][C][V]
Image 4: [C][C][V][V][C]
Image 5: [V][V][C][C][V]

After mosaic():
Result: [V][V][V][V][V]  ← First valid from stack per pixel
```

**Why Mosaic**:
- Satellite imagery has clouds, gaps, sensor limitations
- Mosaicking creates gap-free composite by prioritizing early observations
- Ensures every location has valid data (or flags as missing if all clouds)

---

### 2.6.3 Step 3: Band Selection

**Purpose**: Extract only the 64 AlphaEarth embedding dimensions.

**Code**:
```python
bands = alphaearth_band_names()  # ["A00", "A01", ..., "A63"]
image = image.select(bands)
```

**Why Select Specific Bands**:
- AlphaEarth image has 64 bands (learned embeddings)
- Other satellite data may have hundreds of raw spectral bands
- Selecting only the 64 ensures clean, interpretable output
- Discards raw bands we don't need (e.g., red, green, blue reflectance)

**Band Naming**:
- A00 = 1st embedding dimension
- A01 = 2nd embedding dimension
- ...
- A63 = 64th embedding dimension
- Each represents a learned pattern (e.g., vegetation type, water, urban structure, etc.)

---

### 2.6.4 Step 4: Mean Pooling (Core Aggregation)

#### 2.6.4.1 Conceptual Overview

**Mean pooling** is a spatial aggregation technique that computes the **average value** of a raster image across a geographic region.

**Input**: 
- A satellite image with 64 bands 
- A plot geometry (square polygon)
- Native resolution: 10m pixels

**Output**:
- For each band, ONE scalar value (the mean across all pixels in the plot)
- Result: 64 scalars (one per embedding dimension)

---

#### 2.6.4.2 Visual Example: Mean Pooling in Action

**Scenario**: 2 ha plot with 20×20 pixels (10m each)

```
Plot Geometry (10m × 10m pixels):
┌─────────────────────────────────────────┐
│ Embedding Band A05 (raw 10m pixel values)│
│                                         │
│  0.12  0.15  0.18  0.14  0.16  ...     │
│  0.19  0.21  0.17  0.20  0.18  ...     │
│  0.14  0.16  0.19  0.15  0.22  ...     │
│  0.13  0.18  0.16  0.21  0.19  ...     │
│  0.20  0.17  0.14  0.18  0.15  ...     │
│  ...   ...   ...   ...   ...             │
│ (20×20 = 400 pixel values total)        │
└─────────────────────────────────────────┘

Mean Pooling:
  Sum all 400 values = 72.8
  Divide by count    = 72.8 / 400
  Result             = 0.182 ← Single aggregated value for A05
```

**For All 64 Bands**:
```
Input Image @ 10m resolution:
  Band A00: 400 pixels → mean_A00 = 0.124
  Band A01: 400 pixels → mean_A01 = 0.187
  Band A02: 400 pixels → mean_A02 = 0.045
  ...
  Band A63: 400 pixels → mean_A63 = 0.201

Output:
  64 scalar values (one per band) = 1 plot embedding vector
```

---

#### 2.6.4.3 The Math Behind Mean Pooling

For each embedding band $b \in [A00, A01, ..., A63]$:

$$\text{mean}(b) = \frac{1}{N} \sum_{i=1}^{N} \text{pixel\_value}(b, i)$$

Where:
- $N$ = number of 10m pixels within plot geometry
- $\text{pixel\_value}(b, i)$ = value of band $b$ at pixel $i$

**Example Calculation** (for band A05):

| Pixel | A05 Value |
|-------|-----------|
| 1     | 0.12      |
| 2     | 0.15      |
| 3     | 0.18      |
| ...   | ...       |
| 400   | 0.16      |
| **Sum** | **72.8** |
| **Count** | **400** |
| **Mean** | **0.182** |

#### 2.6.4.4 Earth Engine Implementation

**Code**:
```python
reduced = image.reduceRegions(
    collection=feature_collection,    # Plot geometries
    reducer=ee.Reducer.mean(),        # Mean pooling function
    scale=10,                         # 10m pixel resolution
)
```

**What `reduceRegions()` Does**:

```
For each plot geometry:
  1. Identify all 10m pixels within the plot boundary
  2. Extract values from all 64 bands for those pixels
  3. Apply reducer (mean) to each band separately
  4. Return 64 scalar results
  
Repeat for all plots in the feature collection
```

**Processing Flow**:
```
Input Feature Collection:
  ├─ Feature 1: Plot A (geometry: polygon)
  ├─ Feature 2: Plot B (geometry: polygon)
  └─ Feature 3: Plot C (geometry: polygon)

Each feature enters reduceRegions() pipeline:
  
  Feature 1 (Plot A) →
    [Extract pixels A00-A63 within geometry]
    [Compute mean for A00, mean for A01, ..., mean for A63]
    [Attach 64 means as properties to feature]
    → Output Feature 1 with 64 new mean properties
  
  Feature 2 (Plot B) → [same process] → Output Feature 2
  Feature 3 (Plot C) → [same process] → Output Feature 3

Output: FeatureCollection with properties:
  ├─ Original properties (plot_id, year, lat, lon, etc.)
  ├─ A00_mean: 0.124
  ├─ A01_mean: 0.187
  ├─ A02_mean: 0.045
  └─ ... A63_mean: 0.201
```

---

#### 2.6.4.5 Pixel Alignment & Scale Parameter

**The `scale=10` parameter**:
- Tells Earth Engine: "Resample to 10-meter resolution"
- AlphaEarth is natively 10m resolution, so this matches exactly
- Ensures all pixels are uniform-sized for fair aggregation

**What Happens If Scale Differs**:

| Scale | Effect |
|-------|--------|
| 10m | Matches AlphaEarth native resolution; pixel-accurate |
| 20m | Resamples to 20m (2×2 pixels merged); coarser aggregation |
| 5m | Resamples to 5m (upsampling); data not preserved beyond native |

---

### 2.6.5 Step 5: Why Mean Pooling (Not Other Aggregations)?

#### 2.6.5.1 Alternative Aggregation Methods

**Other options Earth Engine offers**:

```python
# Mean (currently used)
reducer=ee.Reducer.mean()           # Average of all pixels

# Alternative 1: Max
reducer=ee.Reducer.max()            # Brightest pixel (e.g., most vegetation)

# Alternative 2: Min
reducer=ee.Reducer.min()            # Darkest pixel

# Alternative 3: Median
reducer=ee.Reducer.median()         # Middle value (robust to outliers)

# Alternative 4: Standard Deviation
reducer=ee.Reducer.stdDev()         # Variability within plot

# Alternative 5: Percentiles
reducer=ee.Reducer.percentile([25,50,75])  # Distribution shape
```

---

#### 2.6.5.2 Why Mean Pooling is Chosen

| Criterion | Mean Pooling | Justification |
|-----------|--------------|---------------|
| **Interpretability** | ✓ High | Average of plot = representative value |
| **Statistical Properties** | ✓ Good | Unbiased, efficient estimator |
| **Robustness** | ~ Medium | Sensitive to outliers (one bright pixel skews mean) |
| **Computational Cost** | ✓ Low | Simple summation and division |
| **Information Loss** | Moderate | Loses within-plot variance information |
| **Comparability** | ✓ High | Standard across all plots |

---

#### 2.6.5.3 When Mean Pooling Works Well

✓ **Homogeneous plots**: All pixels similar (uniform crop field) → mean is representative

✓ **Temporal averaging**: Annual composite already handles clouds → mean is clean

✓ **Pre-trained embeddings**: AlphaEarth bands designed for downstream tasks → mean is useful feature

✓ **Downstream ML**: Genomic models expect single feature vector per plot → mean provides this

---

#### 2.6.5.4 When Mean Pooling May Fail

✗ **Heterogeneous plots**: Mixed vegetation, buildings, roads → mean loses details

✗ **Cloud remnants**: If mosaic leaves artifacts → mean includes them

✗ **Rare features**: Important but localized pattern → averaged away

---

#### 2.6.5.5 Practical Example: Why Not Max/Min?

**Scenario**: 1 ha plot with one small tree at edge + bare soil

```
Pixel values for Band A05 (vegetation greenness):
  Soil pixels:        [0.02, 0.03, 0.04, 0.02, ...]  (low)
  Tree pixels:        [0.80, 0.82, 0.79]             (high)
  Mostly soil: ~95% of plot, ~5% tree

Mean:    (0.02×0.95 + 0.80×0.05) = 0.07   ← Plot mostly soil (correct)
Max:     0.82                              ← Dominated by single tree (misleading)
Min:     0.02                              ← Dominated by soil (too extreme)
Median:  0.03                              ← Similar to mean (reasonable)
```

Mean makes sense: Plot is mostly soil, so embedding should reflect that.

### 2.6.7 Computational Efficiency: Why Use Earth Engine?

**Why not download raw imagery and pool locally?**

| Aspect | Local Processing | Earth Engine |
|--------|------------------|-------------|
| **Data Size** | 1 year of satellite (TB scale) | Pre-computed embeddings (MB) |
| **Computation** | Hours to days per plot | Seconds per plot |
| **Infrastructure** | Requires servers, storage | Cloud-based (scales automatically) |
| **Maintenance** | Manual updates required | Google maintains imagery |
| **Code Complexity** | Hundreds of lines | ~20 lines in pipeline |

**Speed Advantage Example**:
- Local: Download 100GB imagery, extract, compute = 2 hours
- Earth Engine: Query API, reduce, return = 10 seconds × ~1000 plots = ~3 hours total (but parallel!)

### 2.6.8 Validation: Checking Pixel Counts

**Expected vs. Actual Pixel Counts**:

```
For a 10 ha plot:
  Expected: 10 ha = 100,000 m² / 100 m²/pixel = 1,000 pixels
  
After reduction, check:
  actual_count = pixels_in_geometry_at_10m_resolution
  
  If actual_count ≈ expected_count: ✓ Good (geometry valid)
  If actual_count << expected_count: ⚠ Possible data gap / cloud cover
  If actual_count = 0: ✗ Geometry outside image bounds or all missing data
```

## 3. AlphaEarth Embeddings: Relevance & Significance

### What Are AlphaEarth Embeddings?

**AlphaEarth** is Google's satellite embedding model:
- Trained on multi-year, multi-spectral satellite imagery
- Produces 64-dimensional learned representations of Earth's surface
- Captures spatial patterns at 10m resolution
- Available annually via Google Earth Engine (GOOGLE/SATELLITE_EMBEDDING/V1/ANNUAL)

### Relevance to This Pipeline

| Aspect | Relevance |
|--------|-----------|
| **Spatial Characterization** | Embeddings encode vegetation, soil, infrastructure, and other observable landscape features relevant to crop growth |
| **Environmental Context** | Capture satellite-derived "surroundings" of each plot without manual feature engineering |
| **Temporal Consistency** | Annual images allow year-to-year environmental tracking |
| **Dimensionality Reduction** | 64 dimensions reduce computation compared to raw multispectral bands |
| **Transfer Learning** | Pre-trained embeddings from Google; no training required |
| **Scale Alignment** | 10m resolution matches typical agricultural plot sizes |

### Why Use AlphaEarth Instead of Raw Satellite Data?

| Raw Multispectral Bands | AlphaEarth Embeddings |
|------------------------|----------------------|
| Hundreds of channels | 64 learned dimensions |
| Manual feature engineering needed | Pre-learned features |
| Computationally expensive | Efficient vectors |
| Spectral interpretation required | Spatial patterns encoded |
| Domain-agnostic | Domain-relevant (trained on Earth data) |

## 10. Limitations & Considerations

| Limitation | Impact | Mitigation |
|-----------|--------|-----------|
| **Approximate geometry** | Square ≠ actual plot shape | Validate against known plot polygons if available |
| **Cloud coverage** | Missing data in tropical/cloudy regions | Mosaicking reduces issue; may use seasonal subset |
| **Mean pooling** | Loses within-plot heterogeneity | Could test e.g., stddev, percentile reduction |

| **Year availability** | AlphaEarth started 2017 | Historical plots before 2017 excluded |
| **GCP authentication** | Requires active project & credentials | Ensure `EE_PROJECT` env var set correctly |
| **Plot area errors** | Propagates into geometry error | Quality-check raw CSV for unrealistic areas |

## 12. Relevance of AlphaEarth Embeddings: Summary

### Core Question: "Why AlphaEarth?"

**AlphaEarth embeddings are central because**:

1. **Environmental Phenotyping**: They provide a rich, pre-learned characterization of the environment at each plot—capture what satellite can "see" without manual feature engineering.

2. **Dimensionality & Efficiency**: 64 dimensions reduce computational burden while retaining spatial information (vs. raw multispectral bands with hundreds of channels).

3. **Transferability**: Google's pre-trained model learned features from global satellite data; these patterns transfer well to crop plots.

4. **Standardization**: Same methodology across all plots and years ensures comparability.

5. **Integration**: Bridges genomics (G2F traits) with Earth observation, enabling robust environment-phenotype studies.

6. **Reproducibility**: Fully automated; results are deterministic and easily updated when new satellite data arrives.

- **Predictive models**: Phenotype predictions improve by incorporating satellite-derived environmental features.
